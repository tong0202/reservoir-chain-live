# 双向闭环上真河 · 目标与红线（定死版）

## 一句话
把"水库算 → 链持久存 → 喂回水库"的**双向闭环**整体送上 GitHub 的免费服务器，
手动点火，验证：**免费的河能驱动一个双向闭环真计算一步、可复验、断了能接。**

## 为什么是双向闭环（不是上一版的单向）
- 上一版链（reservoir-chain-live）：水库算延迟 XOR，链只被动记状态 = **单向**。
- 这一版：水库每步算出的**中间状态 s**，写回链持久保存，下一步再喂回水库 = **闭环**。
  链在这里不只是账本，它是那个"不衰减的存储介质"，补上水库 18 步就忘的死穴。

## 任务：带重置的累积奇偶
- `s(t) = s(t-1) XOR u(t)`；若 `reset(t)=1` 则 `s(t)=0`。目标输出 = `s(t)`。
- reset 平均 ~80 步一次，远超水库记忆。要维持的是"算出来的 1-bit 状态"，不是原始输入。
- 本地已证（reservoir_ca_bidir.py）：纯水库 50% 败、放大仍败、双向闭环 100%（含 free-running 自持）。

## 每个 tick 干什么
1. 读上一状态，校验 `self_hash`（篡改则退 2）
2. 恢复水库向量 x、读出层 Wout、**上一步预测状态 s_prev（链持久存的那一位）**
3. 用"创世种子 + 步数"当场派生新输入位 `u_t` 和 `reset_t`（可复验、不可预存答案）
4. 水库吃 `u_t, reset_t, s_prev` 推进一步 → 读出预测 `s_hat`
5. **把 s_hat 写回链的持久态**（下一步喂回自己）= 双向闭环落在链上
6. 父哈希链写新状态

## 红线（防记账，必须守）
- **正确答案（真 parity）从不写进账本。** 只在 tick 结束时临时对照打分。
- 真值靠**重放派生输入序列**当场重算（创世种子 + 步数派生 u/reset，任何人可独立复验）。
- 账本只存：水库状态、Wout、**s_prev（水库自己算的，不是真值）**、预测、命中计数、哈希链。
- 首个 tick 一次性 teacher-forced 训练 Wout（训练时用真 s_prev），之后纯推理 + 自己的预测回喂。

## 边界（不得宣称）
